# Business Requirement: WHOOP Year-Depth History and Body Measurements

**Версия**: 1.0  
**Статус**: Draft  
**Приоритет**: High

---

## 1. Обзор

Расширить WHOOP Service так, чтобы клиент мог:
- запрашивать историю данных с глубиной до одного года;
- получать данные о измерениях тела пользователя из WHOOP API.

Текущий сервис стабильно работает на коротких диапазонах и не предоставляет отдельный endpoint для body measurements.

---

## 2. Проблема (As-Is)

- Для аналитики восстановления и нагрузок нужен более длинный исторический контекст, чем короткие окна.
- В сервисе отсутствует маршрут для чтения телесных измерений (рост, вес, жировая масса и т.д., если доступны в WHOOP).
- Пользователь вынужден обращаться к WHOOP API напрямую, что ломает единый контракт сервиса.

---

## 3. Целевое состояние (To-Be)

- Доступна выборка данных по диапазону до 365 дней для range endpoints (`/cycles`, `/workouts`).
- Добавлен endpoint для получения body measurements в едином контракте сервиса.
- Ошибки по отсутствию нужных OAuth-прав возвращаются предсказуемо через `401 Reauthorization required`.

---

## 4. Функциональные требования

### FR-1: Глубина диапазона до 1 года

**Затрагивает**: `GET /cycles`, `GET /workouts`

**Требования**:
- Поддержать запросы с глубиной диапазона до 365 календарных дней.
- Валидация диапазона:
  - `start` обязателен, timezone-aware;
  - `end` optional (default now), timezone-aware;
  - `end >= start`;
  - `end - start <= 365 days`.
- При нарушении глубины диапазона возвращать `422` с понятным описанием ограничения.

**Ожидания по ответу**:
- Контракт текущих endpoints сохраняется (`status`, `period`, коллекция, `next_token`, `cached`, `timezone_offset`).
- Пагинация продолжает работать через `next_token` в `snake_case`.

### FR-2: Endpoint body measurements

**Новый endpoint**: `GET /measurements/body`

**Назначение**:
- Возвращает актуальные телесные измерения пользователя на момент запроса.

**Источник WHOOP**:
- `GET /v2/user/measurement/body`

**Ответ (200 ready)**:
```json
{
  "status": "ready",
  "measured_at": "2026-03-02T12:10:59Z",
  "height_meter": 1.83,
  "weight_kilogram": 80.4,
  "max_heart_rate": 192,
  "timezone_offset": "+03:00",
  "cached": false
}
```

Примечания:
- Поля, отсутствующие в upstream WHOOP, не возвращаются (omit, без `null`).
- Если в upstream нет данных измерений, возвращать `status="pending"` с reason.

### FR-3: OAuth и права доступа

**Требуемые scopes**:
- Для годовой истории: существующие (`read:recovery`, `read:sleep`, `read:cycles`, `read:workout`).
- Для body measurements: `read:body_measurement`.

**Поведение**:
- При отсутствии нужного scope после auto-refresh и retry сервис возвращает:
  - `401`
  - `reason="Reauthorization required"`.

---

## 5. Нефункциональные требования

### NFR-1: Производительность и устойчивость

- Запросы за 365 дней должны обрабатываться без деградации стабильности сервиса.
- Для range endpoints использовать существующий range-cache c TTL.
- При недоступности WHOOP или timeout возвращать `502` по текущей error-semantics.

### NFR-2: Совместимость

- Existing endpoints и их контракт не ломаются.
- `/week` сохраняется без функциональной миграции.

---

## 6. Критерии приемки

- [ ] `/cycles` и `/workouts` принимают диапазон до 365 дней и корректно отдают данные с пагинацией.
- [ ] Для диапазона более 365 дней возвращается `422` с явным текстом ограничения.
- [ ] Доступен `GET /measurements/body` с данными из `GET /v2/user/measurement/body`.
- [ ] При отсутствии отдельных метрик в WHOOP соответствующие поля опускаются.
- [ ] При отсутствии/утрате нужных прав (`read:workout`, `read:body_measurement`) сервис возвращает `401 Reauthorization required`.
- [ ] Unit/smoke тесты покрывают:
  - валидацию диапазона 365+ дней;
  - happy path body measurements;
  - pending по отсутствию measurements;
  - 401/502 сценарии.

---

## 7. Вне области (Out of Scope)

- Изменение бизнес-логики endpoint `/week`.
- Визуализация body measurements на клиентской стороне.
- Расчёт производных метрик, которых нет в WHOOP upstream.
